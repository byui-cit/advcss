Foundation

Foundation advertises itself as the "most advanced responsive front-end framework in the world". I can't speak to the veracity of that statement, but I do know it is pretty cool.

A framework is really just a collection of code (in this case html, css, and javascript) and design patterns that solve common design problems. Foundation is no exception. It was started by a design company called Zurb. Just like Bootstrap (another front-end framework) it began as a styleguide which was tested and refined and then finally released as an open source project in 2011.

Most projects have the same core set of issues that have to be addressed. One huge advantage of css frameworks (or any framework for that matter) is that they have solutions pre-built and more importantly very well tested to those core problems. Thus they can save developers a lot of time.

Activity

Foundation gives several options for use. The simplest would be to simply download the full, compiled CSS file, and link it into your html normally. This is a great option to get up and going quickly, but is not really great for production. Most sites will not use all of what Foundation offers and so downloading the full CSS is overkill.

The next option would be to take advantage of Foundation's custom download. Using the custom download you can pick and choose which options you want included, and even specify some defaults like colors and fonts.

Foundation can also be quickly integrated into a codepen. If you click on the CSS setting (gear icon next to CSS) then scroll down to the "Add External CSS section" you will see a drop down labeled "Quick-add". Foundation is one of the options there. This can let you quickly try something out.

The best option however is to take advantage of Foundation's Sass version. If you follow the links on the Foundation site for the Sass installation you will eventually end up at a page that gives you several more options just for Sass!

Of the options listed I would recommend either the command line tool (since we have NPM installed) or you could follow the Manual setup: basic template instructions. Both will give you the same result at the end.

On Windows you will probably need to download and install git before you can follow the manual install instructions. Or you could download foundation directly from github - foundation-sites instead of running the git clone command. Then finish following the other intructions.

Once you have it installed you may notice that there are quite a few things in the directory. You may also notice that there is not much in the SCSS directory. Lets talk about the structure that Zurb has given you along with the Foundation Scss files.

Zurb gave you much more than just the scss files you were expecting. The boilerplate that you received is setup similar to what you might find with any medium to large scale modern web project. Modern development projects end up with a lot of moving parts. There are dependencies...other utilities or libraries that are being used to speed up things. Some of those have special requirements as well. Sass would be one example...it needs to be complied before it can be used.

You should notice the first directory is called bower_components and the next file is bower.json. Bower is a package manager. Developers can write reusable modules of code to accomplish specific tasks and then package them up and share them through package managers such as Bower. Those modules are stored in the bower_components directory, and the bower.json contains the configuration. If you look at the contents of the .json file you will see the packages that Zurb needed to work...notice that one of them is the foundation files themselves! You will also see versions. This is one of the greatest strengths of a package manager..they greatly simplify installing and keeping things up to date.

If you did the manual installation you ran a command bower install. It looked at the bower.json file and went and installed all the packages listed. If we needed one of those packages updated to a new version we could simply update the config file then run bower update. Pretty slick huh?

There are also tools that we might need to make everything work. Again the Sass compiler would be a good example. That is governed by another folder and config file: node_modules and package.json. If you open up the .json file again you will see a list of dependencies...the two important ones are gulp and gulp-sass. It determined what happened when you ran npm install

Gulp is a task automator. There can be a lot of steps these days to get your development code ready for production. Compiling Sass to css, running tests, concatinating and minifying dependancies just to name a few. This has made task managers very popular. This one is setup very simply...the only thing it is managing is to run the Sass compiler everytime a change is made (remember the npm start command you were asked to run after the installation?)

Step 01:  To test your installation do the following: Open the app.scss file and take a look. It has a few @imports at the top, then a bunch of @includes. By default almost all of Foundation is included. Lets pretend that we had a project where we only wanted to use the Typography styles from Foundation. Start by commenting out all the @includes. After you save if you check css/app.css you should have a mostly empty file.

Remember that the boilerplate from Zurb comes with a Sass compiler. Make sure you have it running. (open a command line prompt, change directories to your Foundation basic template directory and type npm start)

Uncomment just the lines you think you need to get the Typographic styles.

Step 02:  We can take advantage of the _settings.scss partial to customize our font styles. Lets change the Headline defaults. For our headline font lets use "Palatino Linotype", "Book Antiqua", Palatino, serif. And lets make all headlines Coral in color.
