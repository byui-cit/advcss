SCSS syntax cont.

  1. Sass partials

    Open the link above to the Style Stage starter repo. Browse into the sass directory and review the list of files. Note all of the files that begin with _. These are special Sass files called partials or modules. The leading underscore is important. These files are never intended to be used on their own. Instead they would always be used by being included in a larger SCSS file. The underscore is a message to the Sass compiler to ignore those files until they are used.

    It would never make sense for example to have an about.css file. So we don't want the compiler to ever generate one. The leading underscore accomplishes that.

    Open the style.scss file. Notice that is is the only file that does not have an underscore in it's name. This is the file that we want to be compiled and that we would eventually expect to see a .css version of.

    Note the contents of that file:

    @import "fonts";
// You can modify the theme directly, or override some of the variables
// Overrides just need placed prior to the theme import
// For example:
// $color-primary: #3a0;
@import "theme";
@import "reset";
@import "logomark";
@import "typography";
@import "layout";
@import "header";
@import "nav";
@import "about";
@import "guidelines";
@import "contribute";
@import "files";
@import "styles";
@import "profile";
@import "footer";
@import "links";

    Here we see an example of how to use a partial. This shows the @import syntax, note however that the Sass site is phasing out @import and instead recommends that you use the @use command instead.

    The important thing to note is that if we add another partial file, we should also add another @import or @use line to this file!

    If you open up the public/style.css file in that repo you can see what this file will compile to. That can often help to understand what is going on!

  2. Variables cont.

    If you open up the _theme.scss file next you will see some excellent examples of using not only Sass variables, but also using them together with CSS custom properties.

    At the top of the file you will see the defining of several variables 

    
$color-primary: #3a00ff !default;
$color-primary-alt: #6f6ffd !default;
$color-secondary: #d42257 !default;
$color-background: #d2fbf7 !default;

    You may have noticed the !default keyword. To find out what it does check out the documentation

    After that we see something new.

    $brand-colors: (
    primary: $color-primary,
    primary-alt: $color-primary-alt,
    secondary: $color-secondary,
) !default;

    If you are familiar with lists, Maps, or arrays from other programming languages...this is how you do the same in Sass. If you scroll down to around line 55 of that file you can see an example of using that list. 

    
:root {
    @each $key, $color in $colors {
        --color-#{$key}: #{$color};
    }
    
    --color-anchor: #{$link-color};
    --font-family: "Baloo 2", sans-serif;
    --font-weight-strong: #{$font-weight-strong};
    
    @each $key, $size in $font-sizes {
        --font-size-#{$key}: #{$size};
    }
    
    --border-radius: #{$border-radius};
}

    Here we can see an example of a Sass loop to generate some custom properties. If we were to compile that it would look like this:

  3. Mixin example

    Next open the _logomark.scss file. It's a mixin! Take a moment and look over it. What does it do?

    Next open the _header.scss, and _footer.scss files. Notice a couple of things.   4. The reset

    Finally open up the _reset.scss file. A reset is a CSS file that is designed to remove some or all of the default browser styling. In this case it is a light reset. Review the changes, pay special attention to the last thing in there: 

    
/* Remove all animations and transitions for people that prefer not to see them */
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
